# Course Demo Sync Design

**Date:** 2026-04-16
**Status:** Draft (pending user review)
**Scope:** `dora-moveit2` — aligning the teaching artifacts (`course-script/DoraMoveIt课件_v1.md`, `.pptx`) with the runnable demos in `examples/`, and filling example gaps so students can execute every chapter.

## Context

`course-script/DoraMoveIt课件_v1.md` is an 8-chapter instructor script that walks students from environment setup to a dual-arm pick-and-place demo on `dora-moveit2`. It already exists and is paired with a `.pptx` generated by `course-script/generate_ppt.py`.

`run.md` (added earlier today at repo root) is a copy-paste command index for all 10 demos in the repo. It was written without explicit course alignment.

A review against the course surfaced four issues:

1. **Naming mismatch.** Ch8 tells students to save the final demo as `dual_arm_final_demo.py`, but the actual example file is `examples/dual_gen72/dual_gen72_demo/nodes/dual_arm_example.py`.
2. **Incomplete install list.** Ch1.4 omits `pip install -e dora-mujoco/`, which the dataflows reference via relative path and which `run.md` already installs.
3. **No documented path for student-written scripts.** Ch5/Ch6 show Python code calling `MoveGroup("gen72")` but never explain how to execute that code — the existing dataflow runs its own `user_node`, not user code.
4. **Ch6.4 and Ch6.5 standalone demos are missing.** The course describes a single-arm grasping sequence and a collision-avoidance sequence as standalone lessons, but in the repo they're either absent (grasping is only in the dual-arm script) or buried as one step of five inside `moveit_example.py`.

Separately, Ch8.6 discusses "hardware porting" but there's no dual-arm real-hardware YAML students can point to.

**Goal:** A student who reads the course and runs the commands in `run.md` can execute every referenced demo, including chapters that currently have only inline code. Mismatches between course text and repo artifacts are eliminated.

## Non-Goals

- Regenerating `DoraMoveIt2_课件.pptx` from the updated `.md`. The student runs `course-script/generate_ppt.py` themselves when they want the slide deck refreshed.
- Writing actual Realman SDK code for the dual-hardware template (no hardware or SDK available in this environment).
- Touching the other four example apps (`move_group_demo/ur5e_*`, `hunter_with_arm`, `lekiwi_pick_place`, `nano_pick_place`). The course only uses `move_group_demo` (GEN72 single) and `dual_gen72`.

### Phase 3 — Student-writable custom script workflow

**Create** `examples/move_group_demo/move_group_demo/nodes/my_custom_demo.py` — a heavily commented template:

```python
"""
Student template — write your own MoveGroup script.
Referenced by course Ch5 and Ch6. Run via:
  cd examples/move_group_demo
  dora up
  dora start dataflows/custom_demo_mujoco.yml
"""
from dora_moveit.workflow.move_group import MoveGroup

def main():
    group = MoveGroup("gen72")
    # ============================================================
    # STUDENT: write your code below this line.
    # Examples (see course Ch5 / Ch6):
    #   group.set_named_target("home"); group.go(wait=True)
    #   group.go([1.57, -0.785, 0, -1.57, 0, 0.785, 0], wait=True)
    #   group.set_pose_target([0.15, 0.1, 0.6, 0, 0, 0]); group.go(wait=True)
    # ============================================================

    group.shutdown()

if __name__ == "__main__":
    main()
```

**Create** `examples/move_group_demo/dataflows/custom_demo_mujoco.yml` — user_node points to `my_custom_demo.py`.

**Update** `run.md` with a new "Write your own MoveGroup script" section explaining the edit-then-launch loop (no hot reload — `Ctrl+C` → `dora stop` → re-edit → `dora start` again).
